# Business Understanding — Investment Platform

---

## What Business Problem Does It Solve?

This platform is a **managed investment intermediary system**. It solves the operational problem of running a structured investment program that:

1. Accepts money from investors and puts it to work (investing).
2. Pays investors a guaranteed monthly return (ROI) on their invested capital.
3. Uses a **multi-level agent network** to recruit and manage investors, rewarding agents with tiered commissions.
4. Keeps the operator (Admin) in full control of every financial decision — every deposit, withdrawal, and KYC must pass through Admin approval before money moves.

In short: **it is the back-office and client portal for a private investment fund that distributes business through a network of agents.**

---

## Who Uses It?

There are three types of users, each with their own portal:

| User | Portal | How They Access |
|---|---|---|
| **Admin** | Admin Panel (`/admin/*`) | Created via database seeder; only one admin exists |
| **Agent** | Agent Panel (`/agent/*`) | Created exclusively by the Admin |
| **Investor** | Investor Panel (`/investor/*`) | Registered independently; then linked to an Agent |

---

## Who Are the Customers?

- **The business operator** is the Admin — the company running the investment fund.
- **Agents** are the business's distribution network — sales intermediaries who recruit investors and earn commissions in return.
- **Investors** are the end customers — individuals who deposit money and earn monthly returns on their capital.

---

## User Role Responsibilities

### 1. Admin
The Admin is the sole operator and gatekeeper of the entire platform.

**Responsibilities:**
- Creates all Agent accounts (investors cannot self-sign-up; agents cannot create their own accounts).
- Reviews and approves or rejects KYC document submissions from both Agents and Investors.
- Reviews and approves or rejects all Deposit requests (money coming in).
- Reviews and approves or rejects all Withdrawal requests (money going out), for both Investors and Agents.
- Reviews and acts on Investment close requests from Investors.
- Configures global system settings:
  - Monthly ROI percentage (5%–7%, admin-configurable).
  - Minimum investment amount.
  - Commission percentages per agent level (Levels 1–4).
  - Branding settings (app name, logo, colors).
- Monitors the health of the entire business through dashboards and reports.
- Can reset passwords for any Agent or Investor.
- Can ban/unban any Agent.
- Can impersonate an Agent (log in as that agent to inspect their account).
- Can send targeted notifications to individual Agents.

**What the Admin cannot do:** There is no mechanism for the Admin to directly transfer money to users — all money movements are triggered by user actions and then approved by the Admin.

---

### 2. Agent
Agents are the platform's salesforce. They recruit investors and build downline networks to earn layered commissions.

**Responsibilities:**
- Completes KYC (identity verification) before being able to withdraw commissions.
- Builds a downline: can assign other Agents as sub-agents beneath them (up to 4 levels deep) and can assign Investors directly to themselves (only if they are at Level 4).
- Searches for unassigned users (Agents or Investors) and links them into their network.
- Views their full downline tree — their sub-agents and the investors under those agents, visualized hierarchically.
- Earns commission automatically whenever any investor in their downline (up to 4 levels) makes an investment or a top-up.
- Monitors their commission balance in a wallet dashboard.
- Requests a commission withdrawal once per month (subject to Admin approval).
- Manages their own profile and account settings.
- Views notifications sent by the Admin or generated by system events.

**What an Agent cannot do:** Agents cannot create investors directly — investors self-register and are then linked to an agent. Agents cannot invest themselves. Level 4 agents cannot sponsor further sub-agents.

---

### 3. Investor
Investors are the capital providers. They deposit money, invest it, and earn monthly returns.

**Responsibilities:**
- Completes KYC (identity verification) before investing or withdrawing.
- Deposits funds into their wallet (via a gateway chosen at time of request; Admin approves the deposit).
- Once the wallet is funded, places an investment from their wallet balance.
- Can top up an existing investment at any time — each top-up creates a new investment record and starts a fresh 30-day ROI cycle.
- Earns ROI every 30 days, credited to their wallet automatically.
- Can request to close an investment — this creates a "close request" that the Admin must approve. Upon closure, the initial investment amount plus total earned return is returned to the investor's wallet.
- Can request a wallet withdrawal once per month (subject to Admin approval).
- Manages their own profile, password, and account settings.
- Views their KYC status, investment history, ROI history, and wallet transaction ledger.

**What an Investor cannot do:** Investors cannot refer other investors. They cannot withdraw without Admin approval. They cannot invest without first completing KYC.

## Major Workflows

### Workflow 1: New Agent Onboarding
```
Admin creates Agent account
  → Agent receives credentials
  → Agent logs into Agent Portal
  → Agent submits KYC documents
  → Admin reviews and approves/rejects KYC
  → Agent can now operate (assign users, withdraw commissions)
```

### Workflow 2: New Investor Onboarding
```
Investor self-registers (account exists but unassigned)
  → Agent searches for investor by username in the system
  → Agent (Level 4 only) assigns the investor to themselves
     OR Admin links investor to an agent
  → Investor submits KYC documents
  → Admin approves/rejects KYC
  → Investor is now active and can deposit
```
> **Note:** Only Level 4 agents can directly assign investors. The restriction is enforced in code.

### Workflow 3: Depositing Money
```
Investor selects a payment gateway and enters an amount
  → Deposit record created with status: INITIATED
  → Admin sees the deposit in the pending queue
  → Admin approves → Deposit status = SUCCESSFUL
                   → Investor's wallet balance is credited
  → Admin rejects → Deposit status = REJECTED
                  → No balance change
```
> **Real payment processing is not implemented.** The gateway is a free-text field — the operator processes payment externally and marks the deposit manually.

### Workflow 4: Making an Investment
```
Investor (with funded wallet and approved KYC) places an investment
  → Amount is deducted from wallet balance
  → Amount is added to investment balance
  → Investment record created (type: INITIAL, status: ACTIVE)
  → 30-day ROI cycle begins (nextRoiDate = today + 30 days)
  → Commission distribution runs immediately:
      Level 1 agent (direct referrer) receives their % of the investment
      Level 2 agent (referrer's sponsor) receives their %
      Level 3 agent receives their %
      Level 4 agent receives their %
      (Commission is added to each agent's commissionBalance instantly)
```

### Workflow 5: Investment Top-Up
```
Investor adds more capital to an existing investment
  → Same flow as a new investment (wallet deducted, investment created as TOP_UP type)
  → Fresh 30-day ROI cycle starts for the top-up amount
  → Commission distribution runs again on the top-up amount
```

### Workflow 6: ROI (Return on Investment) Credit
```
Every 30 days from the investment date:
  → ROI is calculated: investedAmount × (monthlyRoiPercentage / 100)
  → ROI amount is credited to the investor's wallet
  → ROI record is created (status: CREDITED, monthIndex increments)
  → Investment's nextRoiDate advances by 30 days
```
> **⚠️ The automated ROI crediting scheduler/cron job is NOT implemented in the current codebase.** The ROI log exists and the service can be called, but there is no background job that triggers this automatically. This must be triggered manually or via a yet-to-be-built scheduler.

### Workflow 7: Withdrawal Request (Investor)
```
Investor requests a withdrawal from wallet balance
  → Amount is deducted from wallet immediately (balance held)
  → Withdrawal record created (type: ROI_WALLET, status: PENDING)
  → Admin sees it in the withdrawal queue
  → Admin approves → Withdrawal status = APPROVED (payment processed externally)
  → Admin rejects  → Withdrawal status = REJECTED
                   → Amount is refunded back to investor's wallet
```

### Workflow 8: Withdrawal Request (Agent Commission)
```
Agent requests a commission withdrawal
  → Amount is deducted from agent's commissionBalance immediately
  → Withdrawal record created (type: COMMISSION, status: PENDING)
  → Admin reviews and approves or rejects
  → On rejection: amount is refunded to agent's commissionBalance
```
> The "once per month" rule described in `project_context.md` is **not enforced in the current backend code**. There is no monthly limit check in the withdrawal controller.

### Workflow 9: KYC Verification
```
User (Agent or Investor) uploads identity document
  → Provides: document type, document number, front image URL, back image URL
  → KYC record created (status: PENDING)
  → User's kycStatus on their profile = PENDING
  → Admin is notified
  → Admin reviews document images
  → Admin approves → kycStatus = APPROVED on both KYC record and User profile
  → Admin rejects  → kycStatus = REJECTED; admin can add remarks explaining why
  → User can resubmit if rejected
```
> **Document images are stored as URLs, not uploaded directly.** The actual image upload mechanism (multer is installed but the KYC controller accepts URLs in the request body) is partially implemented.

### Workflow 10: Investment Close Request
```
Investor requests to close an investment
  → Investment status changes to: CLOSE_REQUEST
  → Admin sees it in the "Close Requests" queue
  → Admin approves closure → Investment status = CLOSED
                           → (investmentAmount + totalReturn) is credited back to investor's wallet
  → Admin can reject the request (investment remains ACTIVE)
```

### Workflow 11: Agent Hierarchy (Downline Building)
```
Admin creates a Level 1 Agent
  → Level 1 Agent can assign Level 2 sub-agents beneath them (Level 1 + 1 = Level 2)
  → Level 2 Agent can assign Level 3 sub-agents
  → Level 3 Agent can assign Level 4 sub-agents
  → Level 4 Agent can assign Investors (their direct clients)
  → Level 4 Agents CANNOT create sub-agents (chain ends at Level 4)
```
The hierarchy is 4 levels deep. Commissions flow up the tree — Level 1 earns the most (on every investor in the entire subtree), Level 4 earns on only their own investors.

     ▼

     │                                                              │

     ▼                                                              │

     ├──► Level 2 agent → commissionBalance +X%                     │

     ▼

## Key Business Constraints (Verified in Code)

| Rule | Enforcement Status |
|---|---|
| Only Admin can create Agents | ✅ Enforced — Agent creation requires `adminAuthMiddleware` |
| Only Level 4 Agents can assign Investors | ✅ Enforced — checked in `assignInvestorController` |
| Agents above Level 4 cannot create sub-agents | ✅ Enforced — level >= 4 check in `assignAgentController` |
| Only Agents can withdraw commissions | ✅ Enforced — role check in withdrawal controller |
| Investors must have wallet balance before investing | ✅ Enforced — balance check in `createInvestmentService` |
| Only pending withdrawals can be approved/rejected | ✅ Enforced — status guard in `updateWithdrawalStatusService` |
| Rejected withdrawal refunds balance | ✅ Enforced — balance is restored on rejection |
| KYC resubmission allowed only after rejection | ✅ Enforced — PENDING or VERIFIED blocks new submission |
| Commission distributed up to 4 levels deep | ✅ Enforced — loop capped at level 4 in `distributeCommissionService` |
| One withdrawal per month per user | ❌ **NOT enforced** — no monthly limit check in current code |
| Automated ROI disbursement every 30 days | ❌ **NOT implemented** — no scheduler/cron exists |
| KYC required before investing or withdrawing | ❌ **NOT enforced in code** — KYC status is stored but not checked at investment/withdrawal time |
| Agents cannot invest | ❌ **NOT explicitly blocked** — the investment endpoint only checks for investor existence, not role |
